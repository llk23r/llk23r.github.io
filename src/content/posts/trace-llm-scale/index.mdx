You type:

> Explain how LLMs work at large scale.

The cursor blinks. For a moment, nothing happens.

Then the answer starts appearing before it is finished.

That detail is the crack in the wall. The machine is not revealing a completed essay. It is choosing a small piece, sending it, choosing the next small piece, sending that, and repeating the loop while many other people are asking their own questions.

So the question is not just "what is an LLM?" The better question is: what has to stay true, at every layer, for that next little piece of text to appear?

This post follows one request. It starts at your keyboard, passes through web machinery, becomes numbered text pieces, runs through learned numbers on GPU chips, chooses one next piece, streams it back, and repeats. Only after the one-request story is clear do we widen the frame to the fleet: the many shared machines making the same trick work for everyone else.

## Q0: Before You Press Enter, What Was Already Waiting?

The most expensive part of this answer happened before you arrived.

Start with one ordinary sentence on some page:

> Paris is the capital of France.

At first it is only text in storage. Before it can teach a model anything, it must pass through a data pipeline: a chain of cleaning steps. Some steps remove broken pages. Some remove repeated copies. Some decide whether the language, topic, and quality belong in the training mixture.

The selected pile is the **training corpus**: the broad body of text the lab intends to learn from. A **dataset** is a more specific prepared version of that material for one training run. The difference matters because the raw internet, the cleaned corpus, and the final dataset are not the same object. A model learns from what survives the pipeline, not from everything humans have ever written.

The model still cannot learn from letters directly. It needs the text cut into pieces. A **token** is one of those pieces: sometimes a word, sometimes part of a word, sometimes punctuation or whitespace. A **token id** is the number assigned to that piece. We will open this mechanism carefully when the request reaches the text-cutting step. For now, keep the object simple: the sentence becomes pieces such as "Paris," "is," and "capital"; those pieces become integer labels; the model works with the labels.

Training asks the same question again and again:

> Given the pieces so far, what piece should come next?

If the model sees "Paris is the capital of", the right next piece is "France." The model does not answer with one word at first. It gives every possible next piece a raw score. A high score means "this looks likely." A low score means "this looks unlikely." Later, when the model chooses the next token, those raw scores get their technical name: **logits**.

Now the training program needs one number that says how badly the model guessed. That number is called a **loss**. Low loss means the model put a lot of belief on the right next piece. High loss means it was surprised by the right next piece.

The next question is stranger:

> If the guess was wrong, which learned numbers should move, and in what direction?

Picture one learned number as a tiny dial. Turn it a little one way and "France" gets a slightly higher raw score. Turn it the other way and "France" gets a slightly lower score. Training has to discover, for billions of these dials, which way each one should move.

A **gradient** is that direction signal. It says, for this learned number, "move up a little" or "move down a little" if you want the loss to shrink. Not every dial matters equally for this sentence. Some get a strong direction. Some barely move.

But the wrong guess happened at the end of the model, while many of the useful dials live deep inside earlier layers. **Backpropagation** is the reverse walk that carries blame from the final miss back through the operations that contributed to it. It is not the model "thinking backward." It is accounting: this raw score came from this intermediate list of numbers, which came from this layer, which came from these earlier weights, so each contributing weight receives its share of the correction signal.

An **optimizer** decides how big the actual nudge should be. The gradient says the direction; the optimizer turns that direction into a careful update so training does not jerk the model around too violently. A **weight** is one learned dial. A **parameter** is the general name for a learned dial like that.

Pause on that loop with one imaginary dial:

| Moment | What changes |
|---|---|
| Before seeing the example | the dial has some current number |
| The model guesses | "France" gets too small a raw score |
| Loss measures the miss | the training run has one number for "how wrong" |
| Gradient points | this dial should move slightly upward |
| Optimizer nudges | the dial moves a tiny amount, not a wild jump |
| Next similar example | "France" may now get a slightly better score |

That is the small version of learning. The model is not memorizing the sentence as a line in a notebook. It is nudging many shared dials so the next-token game works a little better across many similar contexts.

One sentence barely changes anything. Training is the same small correction repeated across huge amounts of text: clean data, cut text into pieces, guess the next piece, measure the miss, move the learned numbers a little, save a checkpoint so the run can resume after failure.

<TrainingStepTrace client:visible />

The toy display above still uses some training words because it is showing the real shape of the loop. Read it left to right: choose the correct next piece, look at the model's raw scores, turn the miss into a loss, compute directions, and nudge the weights. The numbers are tiny on purpose. A real training run does the same kind of loop across many machines.

This is the training mirror of serving. **Serving**, also called **inference**, means running the already-trained model forward to produce an answer for a waiting user. Training asks, "how should the learned numbers change?" Serving asks, "using the learned numbers we already have, what should come next?"

That first trained model still is not necessarily a good assistant. It may be good at continuing text, but a chat product needs more: answer the question, use tools correctly, refuse harmful requests, solve math when possible, cite sources when asked.

So a second family of training steps happens after the base training. This is **post-training**: training after the first big training run, meant to shape behavior.

Trace one post-training example. The example is chosen because it represents behavior the product wants. The model produces possible answers. Those answers are not final user answers yet; they are training material. A human may choose the better answer. A separate scoring model may give a score. A checker may verify whether a math result is correct. A written rulebook may guide critique and revision. That signal becomes another loss. The loss becomes another weight update. The updated **snapshot**, a saved version of the learned numbers, then has to pass evaluations before it can serve users.

<PostTrainingLoop client:visible />

The loop above is still a little too clean. Real post-training is not one single "alignment" operation. It is a sequence of smaller conversions: an example becomes possible answers, those answers become a signal, the signal becomes a loss, the loss becomes a weight update, and the updated snapshot has to survive evaluations before it can be served.

<PostTrainingTrace client:visible />

The names can wait until the path is visible:

| Name | Plain-English shape |
|---|---|
| Supervised fine-tuning, or SFT | show the model instructions and good answers to imitate |
| Reinforcement learning from human feedback, or RLHF | learn from human preferences, then update the model toward answers that score better |
| Direct preference optimization, or DPO | train directly from "this answer was preferred over that answer" pairs |
| Rollout | a sampled answer used as training material, not a final answer to a user |

Other specialized names matter in research papers and infra planning, but they are not needed to understand the first journey. Their shared shape is the important part: examples or preferences become a loss, and the loss changes future weights.

Different labs use different recipes. The invariant is simpler: serving starts with a frozen artifact, not with an open training loop. A deployed **artifact** is the bundle the serving system loads: model weights, tokenizer rules, chat format, safety rules, and configuration files. Your chat may produce logs or feedback for future training, but the answer you are reading is generated by the weights already loaded for this request.

That artifact is not "a file with intelligence inside." It is learned numbers plus exact rules for reading your request and writing the answer back. If the rules and numbers disagree, the model can fail before the first real answer piece is possible.

So when you press Enter, you are not "training the model." You are asking a fixed deployed snapshot to run forward using your current input. The future training loop may learn from aggregate feedback later, after logging, privacy filtering, example selection, labeling, and evaluation. It is not secretly updating the model while you watch the answer stream.

:::note[The Numbers]
| Quantity | Public example |
|---|---:|
| GPT-3 largest model | 175 billion parameters |
| GPT-3 largest context window | 2,048 tokens |
| The Pile public corpus | 825 GiB of English text from 22 subsets |
| Deduplication result in Lee et al. | memorized unprompted output reduced by 10x in their experiments |
| InstructGPT preference result | 1.3B model preferred over 175B GPT-3 on the paper's prompt distribution |
| Transformer paper training run | 3.5 days on 8 GPUs for WMT14 English-to-French |
| Chinchilla training recommendation | train compute-optimal models with far more tokens per parameter than many earlier large models |
:::

:::tip[How We Know]
The transformer architecture was introduced by Vaswani et al., [Attention Is All You Need](https://arxiv.org/abs/1706.03762) (2017). Gao et al., [The Pile](https://arxiv.org/abs/2101.00027) (2020), is a public example of a documented large text mixture, and Lee et al., [Deduplicating Training Data Makes Language Models Better](https://arxiv.org/abs/2107.06499) (2021), measured repetition and memorization effects from duplicate training text. Brown et al., [Language Models are Few-Shot Learners](https://arxiv.org/abs/2005.14165) (2020), describes GPT-3's 175B-parameter autoregressive setup. Hoffmann et al., [Training Compute-Optimal Large Language Models](https://arxiv.org/abs/2203.15556) (2022), is the public Chinchilla scaling-law paper. Ouyang et al., [Training language models to follow instructions with human feedback](https://arxiv.org/abs/2203.02155) (2022), documents the public InstructGPT pipeline and the 1.3B-versus-175B preference result. Later source notes point to public papers on preference training, compact model adaptation, and reasoning-oriented training. The point for now is smaller: this stage creates the learned numbers that serving will later load and run.
:::

If this early stage goes wrong, you notice it later as behavior: stale facts, brittle reasoning habits, unsafe completions, or bad tool-use patterns. Serving can make the model faster or slower. Serving cannot make frozen weights know something they never learned or never receive in the prompt.

This distinction matters for hallucinations. Some hallucinations are training failures: the model learned a pattern of plausible continuation without reliable grounding. Some are prompt failures: the needed fact was not in the context. Some are next-piece choice failures: the model had the right evidence available but still continued in an unsupported direction. Some are system failures: retrieval found the wrong chunk or a tool result was ignored. You diagnose them by locating which part of the journey supplied, distorted, or failed to supply the evidence.

:::note[Explain It Back]
At this point, you should be able to say: training is the long earlier process that nudges learned numbers so the next-piece guessing game improves. Serving is different: it uses a frozen snapshot of those numbers to answer you now. Your chat is not secretly rewriting the model while you watch the text appear.
:::

## Q1: What Leaves Your Browser?

You press Enter.

The browser does not send "your question" by itself. It sends a structured web message called an **HTTP request**. That message has a method such as "send this," a path naming where it should go, headers that act like labels on the envelope, cookies or login proofs that show identity, a body containing the message, and enough metadata for the service to know who you are, what conversation this belongs to, and which product surface you are using.

If the product uses streaming, the client asks for a response that can stay open. That is the important shape: the request does not end as soon as the server starts answering. The connection remains alive so answer fragments can keep arriving.

One common web shape is **Server-Sent Events**. The browser makes a normal request, then the server sends a sequence of small event messages down the response. Another shape is **WebSockets**. In a WebSocket, the browser and server keep a two-way connection open so either side can send messages. The user-visible difference is small: text appears incrementally. The systems difference matters: the server must hold a real connection while the model is still generating.

Before any chat data moves, the browser needs three things. First, it needs to find the machine. **DNS** turns a human name like `example.com` into a network address. Second, it needs a path for the request. **TCP** or **QUIC** creates the transport: the rules for carrying ordered data between machines. Third, it needs privacy and proof of identity. **TLS** checks that the server is really the server it claims to be and creates encryption keys so outsiders cannot read the request. Only after those steps does the chat request cross the internet.

This part looks like normal web infrastructure because it is normal web infrastructure. The full internet stack is a different trace. The LLM-specific reason to care is narrower: the stream has identity, cancellation, ordering, and **backpressure**, which means the receiver cannot safely accept data as fast as the sender could produce it. A request id must follow your prompt into logs and work records. A disconnect must eventually free the model memory reserved for this conversation. A stalled client should not make a GPU keep generating answer pieces forever. The web leg is the control surface for the model leg.

The payload also starts carrying model-specific intent. It says, explicitly or implicitly: this conversation id, this model alias, this maximum output length, this stream preference, this tool set, this safety context, and this cancel handle. Those fields look like product metadata at the browser boundary. A few milliseconds later they become accounting inputs, backend-choice constraints, work-queue state, and memory lifetime.

<RequestEnvelopeTrace client:visible />

This is why the browser boundary is not just "send text to server." If the request says `stream: true`, the backend must preserve a live release path. If it says `tools: search`, the service must choose a model and product loop that can ask an outside tool for help. If it asks for thousands of output tokens, the serving system has to budget future answer pieces before the first one is generated. If the cancel handle is missing or broken, stopping the UI may not stop the expensive backend sequence.

:::tip[How We Know]
TLS 1.3 is specified in [RFC 8446](https://www.rfc-editor.org/rfc/rfc8446) (2018), and HTTP/2 streaming behavior is specified in [RFC 9113](https://www.rfc-editor.org/rfc/rfc9113) (2022). Server-Sent Events are defined by the [HTML Living Standard](https://html.spec.whatwg.org/multipage/server-sent-events.html). The web leg of an LLM request is built from these same public protocols.
:::

:::note[The Numbers]
| Measurement | What it means |
|---|---|
| New TCP connection plus TLS 1.3 full handshake | one TCP round trip plus one TLS round trip before request data |
| Reused HTTP/2 or HTTP/3 connection | no fresh TCP/TLS setup on the request path |
| Streamed response event | an implementation-specific chunk of bytes, not a model concept |
| Human-visible waiting-time target | first answer piece matters separately from the full answer |
:::

If this layer breaks, the error feels boring: DNS errors, TLS errors, expired login, dropped connections, stalled streams. The model may be perfectly healthy and you still see "network error."

## Q2: What Does the Front Door Check?

The request reaches an edge service or API gateway. This is the front door, and front doors are suspicious.

The gateway checks identity first: is this session valid, is this API key real, has this account paid, is the organization allowed to use the requested model? That part is ordinary access control.

Then it asks a different question: how much future work is this request asking the fleet to promise? A short message that asks for a one-sentence answer is cheap. A long document that asks for thousands of output tokens is not. **Quotas** are the limits around that promise: requests per minute, tokens per minute, spend caps, and abuse limits. The gateway also attaches a request id so every later log line can be connected to the same request. It may run input filters or policy checks before expensive serving begins. It may reject obvious abuse before a GPU ever sees a token.

Now make that concrete. Suppose your request contains 8,000 input tokens and asks for a reasoning model. The gateway cannot just count "one request." It estimates input tokens, requested maximum output tokens, hidden reasoning budget if exposed by the product, tool availability, account tier, and region. It checks whether accepting the request would exceed tokens-per-minute, not just requests-per-minute. It also asks a capacity question: can any already-running copy of the model accept this input without running out of the memory it needs to remember the conversation while generating? If not, the honest answer might be a queue, a smaller model, a lower reasoning budget, a different region, or a rejection before any GPU work starts.

<AdmissionControlTrace client:visible />

This is the first place where chip work and memory become product behavior. A long prompt creates a lot of reading work before the first answer piece. A long requested answer creates repeated generation work after that. A reasoning setting may create hidden sampled answers or checking loops before the final visible answer. Every accepted request also asks the service to keep future space aside for remembering the conversation while the answer is being generated.

That is why the front door needs **admission control**. The plain idea is a restaurant host with a kitchen behind them: do not seat more tables than the kitchen can feed. For an LLM service, "tables" are requests, and "kitchen capacity" is GPU time, memory for remembered conversation state, policy budget, tool budget, and queue room. Admission control is the decision about whether the system can safely accept this request now. It is protecting the next minute of the fleet, not merely checking whether a user is allowed through the door.

Trace one gateway decision as a reservation. The request id is born. The account budget is charged in estimated tokens. The input estimate asks for a read-the-input slot. The output limit asks for future generate-one-more-piece turns. The reasoning setting asks for extra hidden work. The tool list asks for permissions and timeouts. The front door asks whether an already-running copy exists with the right weights, enough memory blocks, and a healthy queue. Only after those checks does the request deserve to become model work.

<RequestReservationTrace client:visible />

The key word is **reserve**. Reservation means the system treats future work as real before it has happened. If the gateway admits a request without reserving future answer pieces, a later generation loop may discover that the fleet promised more work than it can finish. If it reserves too pessimistically, GPUs sit idle while users wait outside. Admission control is therefore an early prediction problem: convert a human message into enough estimated compute, memory, policy, and routing state that the rest of the system can keep its promises.

That makes admission control part of correctness. A system that admits too much traffic creates timeouts and retries, which create more traffic. A system that rejects too eagerly wastes expensive GPUs and frustrates users. The gateway is the first control loop in the model's path.

Then comes routing. The product name the user sees is not necessarily a single model file on a single machine. It is more like a sign on a hallway with several rooms behind it. One room may have the old model snapshot. Another may have a new canary snapshot. Another may be in a different region. Another may support tools or images. **Routing** means choosing which backend machine group should handle this request. A router chooses using a mix of:

- requested model and features
- account tier and speed target
- data residency constraints
- current regional load
- already-running copies with the right weights loaded
- reuse of earlier work for repeated input beginnings
- brownout or incident state

This is the first place where "an LLM" becomes "a distributed system."

:::tip[How We Know]
The public literature on production LLM serving uses the same decomposition: front-door admission, request scheduling, model workers, memory management for remembered token state, and distributed GPU execution. If this layer breaks, the cause can be policy as much as machinery. A user can be denied by quota even when GPUs are idle. A router can send traffic to the wrong pool and make the first answer piece slow. A safety precheck can false-positive. A region can have capacity while another region is overloaded because policy says the request cannot move.

## Q3: What Does the Model Actually Receive?

You typed one message. The model usually receives more than that.

A chat product constructs a **prompt**. In everyday use, "prompt" sounds like the sentence you typed. In the serving system, it means the full input sequence given to the model for this turn. It may include a system instruction, developer instruction, tool definitions, safety rules, conversation history, retrieved documents, images converted into model-readable numbers, and your newest message. In an API product, some of these parts are explicit. In a consumer chat product, some are invisible product machinery.

The model does not have memory in the human sense during a plain serving call. The immediate mechanism is simpler: if the model should use something, that thing must be placed where the model can read it. That readable input is the **context** for this turn. If the old conversation is too long, the product may cut off older parts, summarize them, retrieve selected pieces, or ask a different long-context model. Token budget management is therefore a product and systems problem: keep the exact recent turns, summarize older parts, retrieve important facts, and drop material that no longer helps.

Sometimes the answer needs facts that are not safely stored in the model weights: today's policy, a user's private document, a fresh web page. The product can search outside material and paste the relevant evidence into the prompt before the model answers. That is the first-principles shape of **RAG**, short for retrieval-augmented generation: retrieve evidence first, then generate using that evidence. The important part is not the acronym. The important part is that external evidence must physically enter the model's readable context.

That retrieval path has its own small journey. A document is ingested, meaning the system accepts it into the search pipeline. It is split into **chunks**: small passages that can be searched, moved, and quoted.

Chunking is already a judgment call. A whole document is too large to paste into every prompt. A tiny sentence may be easy to retrieve but may lose the paragraph that explains it. A paragraph-sized chunk often preserves more context but spends more of the prompt budget. Before any model answers, the product has already made a reading decision: what size of evidence is worth carrying?

Each chunk may be turned into an **embedding**. Imagine giving every passage a coordinate on a giant meaning map. Passages about the same idea should land near each other even if they do not use the exact same words. An embedding is that coordinate written as a list of numbers. It is not the document. It is a searchable handle for the document passage.

Those handles and the original chunks are stored in an **index**. An index is the search shelf: a structure built so lookup can be fast instead of scanning every document from the beginning. At question time, the system searches the index, reranks possible chunks, trims them to fit the context, and inserts the chosen evidence into the prompt with citations. Different search and ranking methods can do that work; the important first shape is evidence must actually enter the prompt.

Fine-tuning and RAG solve different problems. **Fine-tuning** is extra training. It changes the learned dials so the model develops a habit: a style, a format, a domain behavior, or a tool-use pattern. RAG does not change the dials during this request. It changes what fresh facts are visible on this turn. You can use both: fine-tune a model to follow the company's answer format, then retrieve today's policy document into the prompt.

If tools are available, the prompt contains enough structure for the model to ask product code to do something outside the text generator. That request is a **tool call**. First-principles version: the model writes a structured message like "call search with these words" or "call calculator with this expression." Product code validates the request, runs the outside system, and places the result back into context. Tools reduce hallucination only when the system actually forces external checks into the loop: call the outside API, read the result, quote the retrieved source, and verify the final answer against it. A tool schema, the rule that says what the tool call should look like, does not by itself make a model factual.

If the request includes an image, another front end appears. The image is cut into patches, and those patches are turned into model-readable numbers. A system that combines vision and language this way is called a **vision-language model**, or **VLM**. The exact design varies, but the journey is still the same shape: human content becomes numbered pieces or number tables, then the model operates on those numbers.

<PromptAssemblyPath client:visible />

That component hides one important detail: RAG is two journeys, not one. The first journey happens before your chat, when documents are cleaned, chunked, embedded, and indexed. The second happens during your chat, when the question is searched, possible chunks are found, reranked, trimmed to the context budget, and placed into the prompt. A citation at the end is only trustworthy if every step preserved the evidence: the right document was ingested, the right chunk was retrieved, the reranker kept it, the prompt made it visible, and the model's final sentence actually followed from it.

<PromptEvidenceTrace client:visible />

This is the first-principles test for every "memory," "RAG," "agent," "tool," or "vision" claim: what evidence crossed into the model's actual context, and what did the product force the model to do with it? If the answer is "nothing concrete," the feature is only a suggestion to a generator. **Grounding** means the answer is tied to evidence the system actually supplied. Grounding becomes real when the system retrieves or executes something, validates it, packs it into the prompt, and checks that the final text is supported by that evidence.

<GroundedClaimTrace client:visible />

That last check is the difference between "the model saw a document" and "the answer is grounded." The miniature above uses a toy support check so the failure modes are visible; production systems need stronger citation and policy checks. The retrieved chunk can be wrong. The right chunk can be trimmed out. The tool can return stale data. The model can ignore good evidence and write a plausible sentence anyway. Production RAG systems therefore need observability at the claim level: which source supported this sentence, was that source in the prompt, and did the final answer stay inside what the evidence actually says?

Tools add a second turn inside the turn. The model first emits structured text that means "call this function with these arguments." Product code validates the arguments, calls the outside system, receives a result, and sends that result back into the model as new context. If the tool call is malformed, too slow, unauthorized, or returns surprising data, the product has to decide whether to retry, ask the user, fall back, or stop.

An **agent** is not a different species of model. It is a product loop wrapped around a model. The loop lets the model plan, call tools, read results, update its context, and continue. A multi-agent system splits that loop into roles, but then it must manage shared memory, duplicate work, disagreement, and runaway loops. The first-principles test is simple: does splitting the loop improve the task enough to pay for extra time, cost, and failure surface?

This explains a strange behavior: if something is not in the model weights, not in the current prompt, not retrieved from a tool, and not inferable from those, the model cannot directly use it. There is no hidden notebook it checks unless the product has built one.

## Q4: How Does Text Become Model-Readable Pieces?

The model reads integers.

Before serving, a **tokenizer** maps text into token ids. Think of it as the agreed cutting rule between the product and the model. A token can be a word, part of a word, punctuation, whitespace, a raw storage-sized text fragment, or a common character sequence.

The tokenizer's **vocabulary** is its fixed list of known pieces and their ids. This is not like an English dictionary with definitions. It is more like a lookup table: piece 9125 maps to one row in the model's learned tables. At serving time, the tokenizer must apply the same fixed mapping the model saw during training.

Try changing the text below. The exact ids are toy ids, but the shape is real: text becomes chunks; chunks become numbers.

<TokenizationLab client:visible />

Why not characters? Because character sequences are long. Why not words? Because languages have rare words, new names, code identifiers, emojis, typos, and mixed scripts. Subword tokenization is the compromise: common words can be single tokens; rare words can be assembled from pieces.

After tokenization, the prompt might look conceptually like:

```text
[128000, 9125, 374, 279, 2768, 315, ...]
```

The numbers are indexes into a learned table. Think of the token id as a library card number. It does not contain the book; it tells you which shelf row to pull. The first real model operation is usually an **embedding lookup**: token id 9125 selects row 9125 from an embedding table.

That selected row is a **vector**. Here, a vector is just a list of numbers, but the point is what the list can do. During training, tokens that behaved similarly in many contexts had their rows nudged into useful positions. The row for a token is not a dictionary definition. It is a learned handle the rest of the model can use.

Put many token vectors in rows and you get a **matrix**, a rectangular table of numbers. Now the prompt is no longer text. It is one vector per token position: a table the transformer can transform.

<EmbeddingLookupTrace client:visible />

This is the moment where language turns into table arithmetic. The token id does not "contain meaning" by itself. It selects a row of learned numbers. During training, that row was nudged whenever this token appeared in contexts where it helped or hurt prediction. During serving, the row is just read. The whole prompt becomes a rectangular block of numbers, and every later layer keeps transforming that block while preserving the core shape: one position per token, one vector per position.

:::tip[How We Know]
Subword tokenization for neural sequence models was popularized by Sennrich, Haddow, and Birch, [Neural Machine Translation of Rare Words with Subword Units](https://aclanthology.org/P16-1162/) (ACL 2016). SentencePiece later showed how to train subword models directly from raw text without requiring pre-tokenized whitespace-separated words: Kudo and Richardson, [SentencePiece](https://aclanthology.org/D18-2012/) (EMNLP 2018). GPT-3's model sizes, including 50,257 vocabulary entries, 96 layers, and 12,288 hidden width for the largest model, are listed in Brown et al., [Language Models are Few-Shot Learners](https://arxiv.org/abs/2005.14165) (2020).
:::

:::note[The Numbers]
| Measurement | Example value |
|---|---:|
| GPT-3 largest vocabulary size | 50,257 BPE tokens |
| GPT-3 largest hidden width | 12,288 numbers per token vector |
| GPT-3 largest layers | 96 |
| Prompt length | counted in tokens, not words |
:::

If this layer breaks, the surprise is subtle. A prompt that looks short can tokenize long. A pasted stack trace can explode the context. A multilingual phrase can have very different token counts across tokenizers. A model trained with one tokenizer cannot simply be served with another; the token ids would point at the wrong learned vectors.

The mechanism is worth slowing down for because it explains many surprising costs. In a BPE-style tokenizer, short for **Byte Pair Encoding**, the system starts from small units, often characters or raw byte values. A byte value is one of the small numbers computers use to store text. Then the tokenizer repeatedly applies a learned merge table. Common sequences become single tokens because they appeared often during tokenizer training. Rare names, mixed writing systems, long code identifiers, random-looking encoded blobs, compact machine-readable data, and logs fall back into many smaller pieces. SentencePiece-style tokenizers differ in details, but the production rule is the same: the model and runtime must use the exact vocabulary and id mapping expected by the weights.

<TokenMergeTrace client:visible />

The slider is the missing intuition behind many "why did this cost so much?" moments. A human sees a short identifier or a compact blob. The runtime sees many token positions. Each extra position becomes one more embedding lookup, one more row in the prompt matrix, one more possible position to compare with earlier positions, and often one more remembered item during generation. The tokenizer is upstream of almost every cost and waiting-time number in the rest of the journey.

Tokenization also shapes safety and RAG. Chunking a document by "characters" can create chunks that are wildly different in token length. A safety classifier, a smaller model or rule system that labels risky text, may see text before tokenization and disagree with what the main model sees after hidden prompt assembly. A prompt-injection string, text that tries to manipulate the model through the prompt, can hide in retrieved text that looks compact to a human but consumes enough tokens to push the real evidence out of context. Tokens are not an implementation footnote; they are the accounting unit for cost, context, waiting time, and later comparison work inside the model.

## Q5: What Is Loaded Before the Model Can Answer?

Before the request arrives, the model already exists as **weights**: billions of numbers learned during training.

Training is the expensive offline phase. The simplified base-model objective is: given previous tokens, predict the next token. Repeat that across a massive corpus. Every wrong prediction nudges the weights. After enough examples and enough compute, the weights become a compressed statistical machine for continuing sequences.

That base model is not yet a polished assistant. To become chat-shaped, it is usually adapted with the post-training loop we saw earlier: examples, preferences, safety cases, tool-use examples, or some mixture. Different companies use different recipes, but the product goal is the same: make "next token prediction" behave like "helpful assistant response" under the chat format.

A domain fine-tune can also make a model worse outside the new domain. The human version would be practicing only legal contracts for weeks and then becoming worse at casual email. The model version is called **catastrophic forgetting**: extra training improves one kind of task while damaging behavior the base model had already learned.

Production teams fight that by mixing old and new examples, making smaller updates, freezing parts of the model, and testing old skills before release. Another tool is **distillation**. In distillation, a smaller or specialized model learns from answers produced by a stronger model. The important shape is teacher and student: instead of learning only from raw human-written text, the student learns from a model-generated target that carries some of the teacher's behavior.

At serving time, training is over. The weights are mostly fixed. Your request does not update the model's core weights. It flows through them.

Before that can happen, the artifact has to become a live **replica**: a running copy of the model ready to answer requests. A replica is not a second intelligence. It is one loaded instance of the same snapshot, like one open checkout lane running the same rulebook as the others.

The weight files may be too large for one file or one GPU, so they are stored as **shards**. A shard is a piece of the full model. No shard is useful by itself in the ordinary case; the serving system has to load the right pieces onto the right workers so the whole shape exists again.

Think of a large stage set arriving in crates. One crate has the left wall, one has lights, one has props, one has the floor. Opening one crate does not give you a play. The crew has to check every crate, put every piece in the right place, connect power, test the lights, and only then let the audience in. A model artifact is similar: the files are pieces of a machine, and warmup is the work of turning pieces into a ready machine.

A **tensor** is an organized block of numbers with a shape. A vector is a one-dimensional tensor. A matrix is a two-dimensional tensor. A model weight tensor might say, in effect, "this block has 12,288 rows and 49,152 columns, and each entry uses 16 bits." The shape matters because GPU programs expect exact shapes. If a worker loads the wrong block, the numbers are not merely "a bit off"; the next operation may not know how to multiply them.

A configuration file tells the runtime how many layers exist, which tokenizer to use, how many bits store each number, and how the model is split across machines.

A worker loads the pieces it owns. It verifies checksums, which are fingerprints for detecting corrupted files. It checks tensor names, shapes, and number formats. Then it copies weights from storage to ordinary system memory, then into the memory attached to the GPU. It warms small GPU programs called **kernels**, allocates temporary workspace, reserves room for future generated tokens, and announces "I can serve model snapshot X." Until that warmup finishes, the machine is hardware with expensive storage, not a serving replica.

Read that as a checklist, not as a pile of jargon:

| Question | Why it matters |
|---|---|
| Do we have every piece? | missing weights mean the model is incomplete |
| Is each piece uncorrupted? | a changed number file can silently break behavior |
| Does each piece have the expected shape? | later table math needs exact row and column counts |
| Is each piece on the right worker? | split models only work when every worker owns the right slice |
| Are the GPU programs warmed? | the first user should not pay all setup cost |

<ReplicaWarmupTrace client:visible />

This loading path explains why model deploys are not ordinary web deploys. A stateless web process can start and serve a tiny request quickly. A large model replica may need hundreds of gigabytes of weights moved, verified, split across devices, and resident in GPU memory before it can answer the first token.

Trace one tensor during warmup. A metadata entry says, in effect: this learned table belongs to layer 12, has this number format, has this shape, and starts at this byte position inside this shard file. The worker checks that the tensor belongs to this replica piece, reads the bytes, copies them into CPU memory, then transfers them over hardware links into GPU memory. The runtime may repack the layout so GPU kernels can read nearby numbers efficiently. A bad shape, wrong number format, missing shard, unsupported smaller-number format, or tokenizer mismatch stops the replica before any user token reaches the model.

<ModelArtifactTrace client:visible />

That is why "the model" in production is an artifact contract. The weights, tokenizer, chat template, split plan, smaller-number metadata, runtime kernel support, and evaluation record must agree. If the product sends token id 128006 but the embedding table was built for a different tokenizer, the math can still run while the meaning is broken. If one machine loads the wrong shard, the shape may even fit while the answers degrade. Serving reliability starts before the model answers: it starts by proving that every machine is holding the right bytes in the right layout.

Several failures at this layer look like "the model is down" even though no model math has run. The tokenizer file can mismatch the embedding table. The split plan can mismatch the checkpoint shards. One shard can fail checksum. A runtime can support the weights but not the requested smaller-number format. A cold replica can be healthy but not yet warm, which means routing to it creates a delay before the first answer piece that feels like a broken model.

:::note[Explain It Back]
At this point, you should be able to say: model files are not useful until they become a warm running copy. The system must load every needed piece, check that each piece has the expected shape, put pieces on the right workers, and prepare GPU programs before user requests should be routed there.
:::

## Q6: What Happens Inside One Transformer Layer?

The prompt is now a matrix: one row per token position, many numbers per row.

A **transformer layer** is one repeated block inside the model. It takes the prompt matrix and returns another matrix of the same shape. The shape stays the same so the next layer can accept it, but the numbers inside each row change. Stack many layers, and each token position becomes a richer representation of "what this token means in this context."

Why is there so much matrix multiplication in the first place?

A token vector is a row of numbers. A learned weight matrix is a table of numbers. **Matrix multiplication** is the rule for combining a row of numbers with a table of learned numbers to make a new row. In plain language, the model is asking: "make a new set of features from this old set of features." Do that for every token at once, and the prompt becomes many rows multiplied by the same learned table. This is exactly the shape GPUs are built to run quickly.

One chat-style transformer layer, simplified, does this:

1. Normalize the vectors, meaning rescale their numbers so later math stays stable.
2. Compute **attention**, the step where each position decides which earlier positions matter.
3. Add the attention result back to the original stream.
4. Normalize again.
5. Run a per-position editor: a small stack of learned tables that changes each position separately. This editor is often called a **feed-forward network** or **MLP**.
6. Add that result back too.

Those add-back steps are called **residual connections**. The plain idea matters more than the name: a layer does not replace the stream; it edits it. Do this dozens of times and information can accumulate without every layer having to rediscover everything.

The attention step is the famous part. Start with the human problem: a word often needs earlier words to make sense. In "central bank," the word "bank" should lean toward finance. In "river bank," it should lean toward land beside water. The model needs a way for one token position to ask earlier positions, "which of you matters to me right now?"

To do that, the model creates three new vectors for each position:

- **query**: the question this position asks
- **key**: the label an earlier position offers for matching
- **value**: the information that earlier position can contribute

Those names sound abstract, so keep the library-card picture. The query is what you are searching for. The keys are labels on possible records. The values are the records you will actually read if the labels match. The query for the current token is compared with keys from earlier tokens. The comparison scores are turned into weights that add up to one. Those weights mix the value vectors. That mixture becomes the attention output.

Do two tiny fragments:

```text
river bank
central bank
```

When the current position is trying to understand "bank," earlier pieces offer clues. "river" pulls toward land beside water. "central" pulls toward finance. Attention is the mechanism that gives each earlier clue a strength, then blends the useful clues into the current position's number list.

Here is the same idea without symbols:

- `bank` looking back at `river`: strong match, so the land/water clue matters.
- `bank` looking back at `central`: strong match, so the finance clue matters.
- `bank` looking back at unrelated punctuation: weak match, so almost nothing useful is mixed in.

The model is not reading a dictionary entry for "bank." It is using earlier pieces to change the number row for the current piece so the row fits this sentence.

An **attention head** is one copy of this compare-and-mix mechanism. Models usually run many heads at once so different heads can notice different kinds of relationships. One head might track which noun a pronoun refers to. Another might track the nearby words that set the topic. Another might track punctuation or code structure. Each head is still doing the same job: compare, choose strengths, mix.

<details>
<summary>Implementation bridge: the same attention story in shapes</summary>

You do not need this bridge to keep reading. It is here so later engineering notes do not look like magic symbols.

A **batch** is a group of requests or token positions handled together. Let the hidden width be `H`, the number of token positions be `T`, and the number of attention heads be `A`. The input to a layer is roughly:

```text
X: [batch, T, H]
Wq, Wk, Wv: [H, H]
Q = X Wq, K = X Wk, V = X Wv
reshape Q/K/V into [batch, A, T, head_dim]
```

Each head forms a `[T, T]` score table: every position compared with every earlier position it is allowed to see. This is where a painful scaling rule appears. Double the prompt length and the comparison table can become four times as large before optimized GPU code reduces memory traffic.

The compact formula for one attention head is just this sequence:

```text
compare queries with keys
scale the scores so they stay numerically tame
block any future positions
turn the allowed scores into mix weights
mix the value vectors using those weights
```

That is the same compare-and-mix story in implementation order. `QK^T` is the compact notation for comparing every query with every key. Dividing by `sqrt(d_head)` keeps scores from becoming too large as vectors get wider. The mask blocks future positions. The "turn scores into mix weights" step uses a function introduced later when the model chooses an output token.

</details>

If the symbols blur, keep the non-symbol version: every position asks a question, earlier positions offer labels and information, the model scores the matches, and the useful earlier information is mixed back into the current position.

The "cannot look right" rule is called a causal mask. While generating token 42, the model may use tokens 1 through 41. It may not use token 43 because token 43 does not exist yet.

This is also why decoder-only models became common for chat LLMs. **Decoder-only** means the model keeps extending one sequence from left to right. That shape matches the training game from the beginning: given the previous tokens, predict the next token.

Position still matters. Without position information, attention sees a bag of token vectors, not an ordered sentence. Different models encode position in different ways. For this journey, the important rule is simpler: the model must know not only which tokens exist, but where they sit relative to each other.

There are also variants of how keys and values are shared. The production reason is memory. If many attention heads can share some key and value data, the remembered state for long conversations gets smaller. The tradeoff is direct: less remembered data can mean lower memory use and faster generation, at some quality risk.

The **MLP** is less narratively famous than attention but computationally huge. If attention is how positions talk to each other, the MLP is how each position thinks privately after receiving context. It applies learned table operations and gates independently at each token position. A **gate** is a learned control that decides which intermediate features pass through strongly and which are damped. If attention moves information between positions, the MLP transforms information inside each position.

Some models replace one big MLP with a **mixture of experts**. Think of a workshop with many specialist benches. For each token, a small router chooses a few benches to work on it. Only the selected experts run, so the model can contain many weights without using all of them on every token. The cost is new routing and load-balancing problems, especially when experts live on different GPUs.

Deep stacks need stability tricks. LayerNorm and RMSNorm are two common ways to keep vector scales controlled. Residual connections let layers edit a stream instead of rewriting it.

At the final layer, the model turns the last token's vector into one raw score per vocabulary token. These scores are **logits**. High logit means "more likely next token." Low logit means "less likely." The model has not chosen a word yet. It has produced a score landscape over possible next tokens.

:::tip[How We Know]
The query-key-value attention mechanism and stacked transformer architecture are described in Vaswani et al., [Attention Is All You Need](https://arxiv.org/abs/1706.03762) (2017). Decoder-only language modeling at scale is described in Brown et al., [Language Models are Few-Shot Learners](https://arxiv.org/abs/2005.14165) (2020). Later public papers study position-handling, key/value sharing, and expert routing; useful examples are Su et al., [RoFormer](https://arxiv.org/abs/2104.09864) (2021), Ainslie et al., [GQA: Training Generalized Multi-Query Transformer Models from Multi-Head Checkpoints](https://arxiv.org/abs/2305.13245) (2023), and Fedus, Zoph, and Shazeer, [Switch Transformers](https://arxiv.org/abs/2101.03961) (2021).
:::

Failure at this level looks like model behavior: hallucination, wrong reasoning, instruction mistakes, repetition, over-refusal, under-refusal. The GPU may have executed perfectly. The network may have streamed perfectly. The selected continuation can still be wrong.

At this point, you should be able to say: a transformer layer edits one row per token. Attention lets a row pull useful information from earlier rows. The MLP then edits each row privately. After many layers, the final row is ready to produce raw scores for possible next pieces.

## Q7: Why Remember the Earlier Pieces?

Imagine generating a 500-token answer.

Naively, for token 1 you run the whole prompt. For token 2 you run the whole prompt plus token 1. For token 3 you run the whole prompt plus tokens 1 and 2. That repeats work. The earlier tokens do not change. Their keys and values do not need to be recomputed every time.

So serving systems keep a **KV cache**. The name is short for "key/value cache," but the intuition is simpler: while reading the prompt, each layer computes reusable notes about earlier tokens. During generation, those notes let the model look back without recomputing the whole past from scratch.

The first phase, **prefill**, is the big read. It reads the whole prompt and writes those reusable notes into the cache. The second phase, **decode**, is the one-token loop. It generates one token, adds one new set of notes for that token, then uses the enlarged cache for the next token.

<TransformerPass client:visible />

<AttentionKvCache client:visible />

This is why **first-token latency**, the waiting time before the first answer token appears, and tokens-per-second are different metrics.

First-token latency includes queueing, tokenization, routing, and prefill over the whole prompt. After that, each decode step is smaller, but it is sequential: token 257 depends on token 256. You cannot fully parallelize one user's generated tokens across time because the future token needs the past token as input.

The KV cache is also why long contexts are expensive even after prefill. Every active request carries memory that grows with:

```text
number of active tokens
× number of layers
× 2 stored tables per layer
× width of each stored table
× bytes used for each number
```

The `2` is for keys and values.

<KvCacheTrace client:visible />

This is the concrete trade. The cache saves recomputation by remembering the past, but it makes every active sequence occupy GPU memory until the sequence finishes or is cancelled. Some models reduce that pressure by storing fewer key/value tables. **Quantized KV cache** means storing those remembered numbers with fewer bits. It attacks the part of memory that grows with context and concurrency, not just the fixed weight files.

:::tip[How We Know]
The serving literature treats KV-cache memory as a central bottleneck. The vLLM paper, [Efficient Memory Management for Large Language Model Serving with PagedAttention](https://arxiv.org/abs/2309.06180), describes how KV cache can consume a large share of GPU memory and introduces block-based paging to reduce waste. NVIDIA TensorRT-LLM documents production features such as [paged KV cache and in-flight batching](https://docs.nvidia.com/tensorrt-llm/).
:::

:::note[The Numbers]
| Factor | Effect |
|---|---:|
| KV tensors per layer | 2: keys and values |
| KV memory scaling with context | linear in active token count |
| KV memory scaling with concurrency | linear in active sequence count |
| vLLM reported throughput improvement | 2-4x over FasterTransformer and ORCA at similar latency in its evaluated workloads |
:::

If this layer breaks, the problem looks more like operating systems than "AI." If KV memory fills, the server must reject, queue, evict, swap, recompute, or route elsewhere. If memory becomes fragmented, you can have free space in total but no suitable blocks for the next sequence.

:::note[Explain It Back]
At this point, you should be able to say: the KV cache is saved work. The first read of the prompt writes reusable notes. Each later one-token step adds one more note. That makes generation faster, but every active conversation now occupies GPU memory until it finishes or is cancelled.
:::

## Q8: Where Does the GPU Actually Spend Time?

By now the request has become number tables. The next question is physical: where do those numbers spend their time?

Some time is spent doing arithmetic. A GPU has many arithmetic circuits. Some of the most important ones for LLMs are **tensor cores**: circuits built to multiply small number tables very quickly.

The usual unit for arithmetic work is a **FLOP**, a floating-point operation. Floating point is a computer format for decimal-like numbers. One multiplication is a FLOP. One addition is a FLOP. A large matrix multiply contains enormous numbers of them. FLOPs are the "how much math" count.

But arithmetic is only half the story. Before a number can be multiplied, it has to arrive.

A **bit** is the smallest stored value, 0 or 1. Real memory is built from physical devices that hold those 0s and 1s.

The memory hierarchy exists because the best place to keep a number depends on what you need next. **DRAM** is dense memory that stores bits as charge and must be refreshed. GPU **HBM**, high-bandwidth memory, is a form of DRAM packaged close to the GPU so large streams of numbers can move quickly. **SRAM** is faster memory on the chip itself, used for caches, registers, and shared memory, but there is much less of it.

So the trade is not mysterious: far and large is cheaper, close and tiny is faster. An LLM kernel is often a choreography for moving the right number from the large place into the fast place just before the arithmetic circuit needs it.

So the chip is always playing a small logistics game: pull numbers from HBM, keep the most useful chunks on chip, multiply them, accumulate results, and write back only what must be saved. If the numbers arrive late, the arithmetic circuits wait. A "fast" GPU can still be slow if most of its time is spent waiting for memory.

Trace one attention tile. A tile is a small block of a larger number table. A block of GPU threads receives a slice of queries and a slice of cached keys. The keys are too large to keep all on chip, so the kernel streams one tile from HBM into faster on-chip storage. Tensor cores multiply query tiles by key tiles. The kernel keeps only the running summary it needs, then streams the matching value tile and mixes it into the output. Then it moves to the next tile.

If the tile is chosen well, the chip reuses nearby numbers and avoids writing a giant temporary table to HBM. If the tile is chosen poorly, the chip spends its time fetching and storing instead of multiplying. This is why the same attention math can feel fast or slow depending on the memory path.

<KernelRooflineTrace client:visible />

The slider version is intentionally simplified, but the invariant is real. A matrix multiply has arithmetic work and memory traffic. **Arithmetic intensity** means how much math you get for each byte moved. High intensity keeps tensor cores busy. Low intensity makes the kernel wait on memory. This is why practical runtimes care about tiling, combining steps, number layout, smaller number formats, cache locality, and avoiding unnecessary reads and writes.

## Q9: How Can a Too-Large Model Fit?

Start with memory.

If a model has 70 billion parameters and each parameter is stored in 16 bits, the raw weights need about 140 GB in decimal units, about 130 GiB in binary units. GiB is a computer-memory unit close to GB but counted in powers of two. An 80 GB GPU cannot hold that model alone. Add KV cache, temporary workspace, leftover gaps from memory allocation, communication buffers, and safety margin. Now you need multiple GPUs or lower precision.

<GpuMemoryBudget client:visible />

Work the memory budget in the order the runtime experiences it. First come weights: fixed cost, paid before users arrive. Then runtime workspace: temporary buffers, warmed GPU programs, communication buffers, allocator slack, and fragmentation. Then KV cache: small for one short request, enormous for many long ones. Finally safety margin: the system must survive bursty arrivals, uneven sequence lengths, and one request asking for far more output than expected. If the model is split across four GPUs, the weight shard per GPU may fit, but every GPU still needs its share of KV cache and communication workspace. "Fits the weights" is therefore not the same as "serves production traffic."

<ReplicaPlacement client:visible />

That second view is the useful correction. It is not enough to ask whether the weights fit. Ask what lives in HBM at the moment one decode step launches. The answer includes the GPU's share of weights, the remembered key/value data for active requests, temporary workspace, communication buffers, leftover allocator gaps, and emergency headroom. If one GPU in the group runs out of room, the whole running copy is in trouble.

Now trace the ownership question. **Tensor parallelism** means splitting one big matrix multiplication across GPUs. If a 4-way tensor-parallel replica owns one layer's projection matrix, GPU 0 may hold columns 0-2047, GPU 1 columns 2048-4095, GPU 2 columns 4096-6143, and GPU 3 columns 6144-8191. The prompt vector may be copied to all four GPUs, but the weights are not. Each GPU computes the part it can compute locally. The next operation either needs those parts gathered together or reduced into a shared result. That is where "fits across GPUs" becomes "waits across GPUs."

<DeploymentPlanTrace client:visible />

The deployment plan above is deliberately mechanical. Pick a parameter count. Pick a precision. Pick how many GPUs split the model. Then ask what every GPU must hold at the instant a decode iteration launches. The answer is not just the weight shard. It is the weight shard plus KV pages for active sequences, temporary workspace, communication buffers, allocator slack, and enough headroom for bursty prompts. This is where a production plan differs from a model-card number: "70B" is not a deployment; per-GPU memory, interconnect, cache layout, and traffic shape are the deployment.

There are several levers.

The first lever is to make each number smaller. A weight stored in 16 bits takes twice as much memory as a weight stored in 8 bits. **Quantization** is the craft of storing numbers with fewer bits while trying to preserve the model's behavior. It is like rounding measurements on a map: round too gently and you save little space; round too aggressively and roads no longer line up. 8-bit weights roughly halve memory versus 16-bit. 4-bit weights roughly quarter it. This can improve cost and fit, but the system must preserve quality and speed.

The practical question is what gets quantized. Weight-only quantization reduces model memory. Quantizing intermediate numbers can speed or shrink temporary work but is harder to preserve accurately. KV-cache quantization attacks the memory that grows with context and concurrency. Quantization is not just a file-size trick; it changes which kernels run, how memory bandwidth is used, and sometimes which quality failures appear.

The second lever is to split one big operation across chips. **Tensor parallelism** splits individual matrix multiplications across GPUs. Each GPU owns a shard of the weights. During the layer, GPUs exchange partial results.

The third lever is to split the stack of layers. **Pipeline parallelism** puts different layers on different GPUs. Token representations flow through stages like an assembly line.

The fourth lever is to make more copies. **Data parallelism** replicates the whole model on multiple GPU groups. Different requests go to different replicas.

Those three levers combine during training and serving, but the shape of the pain differs. Training stores extra information so it can change the weights later. Serving is trying to answer waiting users with fixed weights, unpredictable request lengths, and a KV cache that grows while streams stay open.

:::tip[How We Know]
Megatron-LM describes tensor and pipeline model parallelism for large transformer training in Shoeybi et al., [Megatron-LM: Training Multi-Billion Parameter Language Models Using Model Parallelism](https://arxiv.org/abs/1909.08053) (2019). ZeRO describes optimizer-state sharding in Rajbhandari et al., [ZeRO: Memory Optimizations Toward Training Trillion Parameter Models](https://arxiv.org/abs/1910.02054) (2019). The later Megatron scaling paper studies tensor, pipeline, and data parallel tradeoffs in [Efficient Large-Scale Language Model Training on GPU Clusters](https://arxiv.org/abs/2104.04473) (2021). Public quantization references include Frantar et al., [GPTQ](https://arxiv.org/abs/2210.17323) (2022), and Lin et al., [AWQ](https://arxiv.org/abs/2306.00978) (2023).
:::

:::note[The Trap Door]
"Use more GPUs" is not a complete answer. More GPUs add memory, but they also add communication. Tensor parallelism can make one model fit, but every layer may now need collective operations. Pipeline parallelism can improve fit, but bubbles reduce utilization. Data parallelism scales throughput, but not the memory needed for one replica.
:::

## Q10: What Crosses Between GPUs?

If the model is split across GPUs, a token vector cannot stay on one chip.

Follow one layer in tensor parallelism. The token representation enters the layer split or copied across a group of GPUs. Each GPU owns a slice of a projection matrix. It multiplies the input by its slice. Now each chip has only part of the answer.

The next operation cannot pretend the partial result is complete. The chips need a coordinated "everyone exchange your pieces now" moment. That is a **collective operation**: a communication step where the whole GPU group participates together. In an **all-reduce**, every GPU contributes partial values and every GPU receives the combined result. In an **all-gather**, every GPU contributes a shard and every GPU receives the assembled tensor. The exact collective depends on how the matrix was split.

That collective is a synchronization point. The fastest GPU waits for the slowest one. The layer cannot advance until the required tensor shape exists.

<TensorParallelTrace client:visible />

The important detail is not the name of the collective. It is the shape contract. Before the layer, the runtime has a tensor with a known shape. Splitting the matrix lets several GPUs compute pieces of the next tensor. But the following operation expects a complete or consistently sharded tensor. The collective restores that contract. If GPU 2 is late, GPU 0 cannot responsibly guess its missing slice. Distributed serving is therefore full of tiny barriers hidden inside what looks like one forward pass.

This is why high-end serving boxes are not just "GPUs in a case." The **interconnect**, the hardware link between GPUs, is part of the model's execution path. In an H100 HGX system, NVLink and NVSwitch let GPUs communicate at far higher bandwidth than ordinary PCIe. Across machines, systems use InfiniBand or high-speed Ethernet, and now network waiting time enters the inner loop for multi-machine serving or training.

Pipeline parallelism moves a different kind of data: the stream of intermediate token vectors. GPU group 0 runs early layers, group 1 runs later layers, and small batches flow through. This can make giant models fit, but it creates bubbles: times when a stage waits because the previous or next stage is not ready.

Expert parallelism appears in mixture-of-experts models. A router chooses a small number of experts for each token, and tokens must be dispatched to the GPUs holding those experts. That improves parameter count per token-compute, but it adds an all-to-all communication problem.

<DistributedSync client:visible />

Training adds another synchronization path. In data parallel training, each replica computes gradients on different data. Before the optimizer step, replicas must agree on the combined gradient or an equivalent sharded form. ZeRO changes what is stored where: optimizer states, gradients, and parameters can be partitioned so each worker holds less redundant state, but the missing pieces must be gathered or reduced at the right moments. Memory saved becomes communication scheduled.

The useful mental model is not "parallelism makes it faster." It is: parallelism trades local memory and compute pressure for communication and synchronization pressure. The right answer depends on model size, batch size, sequence length, how the GPU links are arranged, precision, pipeline bubbles, expert load balance, and the waiting-time target.

:::tip[How We Know]
Megatron-LM introduced practical tensor and pipeline model parallelism for large transformers in [Training Multi-Billion Parameter Language Models Using Model Parallelism](https://arxiv.org/abs/1909.08053). NVIDIA's [NCCL collective documentation](https://docs.nvidia.com/deeplearning/nccl/user-guide/docs/usage/collectives.html) specifies operations such as all-reduce and all-gather. NVIDIA's [HGX H100 platform description](https://developer.nvidia.com/blog/introducing-nvidia-hgx-h100-an-accelerated-server-platform-for-ai-and-high-performance-computing/) describes eight H100 GPUs connected through NVSwitch, with each H100 communicating with the others at 900 GB/s bidirectional NVLink bandwidth.
:::

|---|---:|
| HGX H100 GPUs per baseboard | 8 |

If this layer breaks, the answer may still be correct but slow. A single slow GPU, a flaky link, or an imbalanced expert route can slow the whole group. Distributed inference turns hardware tail latency into user-visible token latency.

## Q11: What Changes When Many People Chat at Once?

One user wants low **latency**: low waiting time for their own answer. The provider wants high **throughput**: many useful tokens served per second across everyone.

These goals fight. Think of a bus. Leaving immediately is great for the first passenger, but wastes seats. Waiting until the bus is full uses the bus well, but makes the first passenger wait. A GPU has the same tension. A nearly empty GPU may give one user a quick answer but waste expensive hardware. A packed GPU may use hardware well but make one user's stream wait.

The conflict is the one-token loop.

For one request, decode generates one token at a time. That is too little work to keep a large GPU busy. The trick is to generate the next token for many requests in the same model step. Your request gets one row in the step. Other people's requests fill the other rows.

But people do not arrive in neat groups. One person sends a short question. Another pastes a long document. One answer ends after ten tokens. Another keeps going for a thousand. A fixed batch would quickly fill with holes.

So modern LLM serving rebuilds the batch while the work is already moving. That is **continuous batching**, also called **in-flight batching**. The batch is not a sealed box. It is a moving roster: requests enter, advance, finish, or cancel between model steps.

A **scheduler** is the part of the runtime that chooses which active requests run in the next model step. The first-principles job is packing: choose a set of conversations that can fit together in compute time and memory right now. It is not merely "first come, first served." It has to protect smooth streams, avoid wasting GPU time, and make sure KV-cache memory does not fill.

<BatchingSimulator client:visible />

This is the systems heart of large-scale LLM serving. The unit is not "run one request to completion." The unit is "advance this set of active conversations by one step."

<SchedulerState client:visible />

A useful mental model is that every active conversation has a small record attached to it. The record says: how long the input was, how many tokens have been generated, whether it is still reading the prompt or already decoding, where its KV-cache pages live, how many tokens it may still produce, and whether the user cancelled. The scheduler repeatedly asks: which records can fit in the next step without breaking compute and memory budgets?

<SchedulerTickTrace client:visible />

Now **continuous batching** has teeth. A cancelled stream is not just a user leaving; it is KV pages that should be freed. A huge prompt is not just a slow request; it may need to be read in chunks so it does not freeze dozens of active answer streams. A high-priority decode is not one more row in a queue; it is a visible next token waiting on this iteration.

Some systems go further and separate **prefill** from **decode** onto different worker pools. Prefill likes large prompt-reading work. Decode likes steady one-token steps that reuse KV cache. Separating them can improve hardware use, but it creates a new problem: the remembered KV-cache state, or enough information to continue generation, must move, be shared, or be rebuilt.

The main idea is simpler than the framework names: large-scale serving is a live packing problem. Pack enough conversations together to keep the chips busy. Do not pack so much that memory fills, streams stutter, or cancelled work keeps consuming space.

ORCA introduced iteration-level scheduling for transformer serving and showed why request-level scheduling wastes capacity for generative models: Yu et al., [ORCA](https://www.usenix.org/conference/osdi22/presentation/yu) (OSDI 2022). TensorRT-LLM documents [paged attention, in-flight batching, and request scheduling](https://nvidia.github.io/TensorRT-LLM/features/paged-attention-ifb-scheduler.html). vLLM's PagedAttention paper connects batching efficiency to KV-cache memory management, and its [PagedAttention design documentation](https://docs.vllm.ai/en/stable/design/paged_attention/) describes sequence, block, thread-group, and KV-cache layout concepts. Hugging Face documents [Text Generation Inference](https://huggingface.co/docs/text-generation-inference/index). Zheng et al., [SGLang](https://arxiv.org/abs/2312.07104) (2023), describes RadixAttention for prefix reuse in structured LLM programs.

Under load, things can change suddenly. Everything is fine until KV cache fills or queues cross a threshold. Then first-token waiting time rises, users keep connections open longer, active conversations accumulate, memory pressure worsens, retries add traffic, and the system can spiral. Rate limits are not just business rules; they are stability controls.

## Q12: How Is the Next Token Chosen?

After the final transformer layer, the model produces logits: one raw score for every token in the vocabulary.

The simplest choice is **greedy decoding**: pick the highest-scoring token. That is easy to understand, but it can make writing brittle or repetitive because the model always takes the same most obvious step.

Chat models often **sample** instead. Sampling means choosing randomly, but not equally. Imagine a raffle where better tokens get more tickets. "France" might get many tickets after "Paris is the capital of"; "banana" might get almost none. A **probability** is the fraction of the raffle tickets owned by one outcome. A **probability distribution** is the full ticket count across all possible next tokens.

The model's raw scores are not ticket counts yet. They can be negative, huge, or tiny, and they do not add up to anything meaningful. The runtime needs a conversion step.

First imagine a simpler rule: give each token a number of raffle tickets based on its score, then divide by the total number of tickets. The highest-scoring token gets the biggest share, but lower-scoring tokens may still get a few tickets. The real rule used most often is **softmax**. Softmax turns raw scores into a probability distribution: higher scores get much more probability, lower scores get much less, and the final probabilities add up to 1. The important idea is not the formula. It is the conversion from "unbounded model scores" to "a raffle the runtime can draw from."

Then the product can shape the raffle before drawing:

- **temperature** changes how sharp the raffle is; low temperature is like giving the favorite almost all the tickets, while high temperature lets runner-up tokens keep more tickets
- **top-p sampling** first draws a circle around the most likely tokens until their ticket shares add up to a chosen threshold, then throws away everything outside that circle before sampling
- repetition penalties remove some tickets from tokens that would repeat too much
- safety or format rules can mask invalid tokens, meaning remove their tickets entirely before sampling

Production code computes softmax carefully because raw scores can be large. It often subtracts the largest raw score before exponentiation. **Exponentiation** means using a number as a power, which can create very large values. Subtracting the largest score does not change the final probabilities, but it prevents the computer's number range from overflowing.

<SamplingTrace client:visible />

Try reading the table as a raffle machine. The raw score says how attractive a token looked. Softmax turns attractiveness into ticket share. Temperature changes whether the favorite dominates or the table stays more open. Top-p removes the long tail of unlikely tokens. The final sample is one draw from the remaining tickets.

:::note[Running Toy Request]
Our toy request has reached the choosing point:

- `Rainbows` likely gets many tickets.
- `They` may get some tickets.
- `Banana` should get almost none.
- Unsafe or invalid pieces may have their tickets removed entirely.

The model has not written the answer yet. It has produced a shaped raffle for exactly one next piece.
:::

Then the chosen token is appended to the sequence.

That last sentence is the whole loop. Append the token, update the KV cache, run the model again, sample again, stream again. Stop when the model emits an end token, hits a length limit, triggers a tool call, violates a policy, or the client disconnects. Greedy decoding, temperature, top-p, masks, and penalties all live at this narrow point: the final vector has already become logits, and the runtime must choose exactly one next token while preserving product constraints.

:::note[Explain It Back]
At this point, you should be able to say: logits are raw scores, not words. Softmax turns scores into ticket shares. Temperature and top-p reshape the ticket pool. Sampling draws one piece, appends it, and the whole model runs again for the next piece.
:::

:::tip[How We Know]

Faster variants such as speculative decoding are described in Leviathan et al., [Fast Inference from Transformers via Speculative Decoding](https://arxiv.org/abs/2211.17192) (2022), Chen et al., [Accelerating Large Language Model Decoding with Speculative Sampling](https://arxiv.org/abs/2302.01318) (2023), and Li et al., [EAGLE](https://arxiv.org/abs/2401.15077) (2024). OpenAI's [Learning to reason with LLMs](https://openai.com/index/learning-to-reason-with-llms/) and Guo et al., [DeepSeek-R1](https://arxiv.org/abs/2501.12948), are public examples of the shift toward extra inference-time work for harder reasoning. Snell et al., [Scaling LLM Test-Time Compute Optimally can be More Effective than Scaling Model Parameters](https://arxiv.org/abs/2408.03314) (2024), studies when spending more test-time compute can beat simply using a larger model.

:::note[Speculative Decoding in One Breath]
**Speculative decoding** is a speed trick. A small draft model proposes several future tokens. The large target model checks them in parallel. If the draft was right, several tokens are accepted for roughly one expensive target pass. If it was wrong, the system falls back without changing the large model's probability distribution, depending on the exact algorithm.
:::

Now add the reasoning-model complication: not every model is trying to minimize tokens. Reasoning models often spend extra serving-time compute before the final answer. The extra work might be hidden scratch tokens, visible reasoning-like text, multiple sampled answer attempts, verifier passes, code execution, search, or tool calls. The product may show none of that internal work and stream only the final answer.

<TestTimeCompute client:visible />

This changes the serving problem. A normal chat request is already variable length. A reasoning request is variable length twice: the final answer length varies, and the amount of hidden or auxiliary work varies. That means the scheduler must reserve enough token budget, KV cache, tool time, and safety checks for work the user may never see. "Reasoning" at product level becomes "more one-token-at-a-time iterations, more verification, and more queue occupancy" at infrastructure level.

Now compress the whole machine into one concrete token.

<SingleTokenTrace client:visible />

This is the closest we can get to the whole post in one object. One visible token is not one operation. It is one scheduler decision, one batch slot, one last-token vector, dozens of layer passes, many GPU-memory reads, many specialized matrix-multiply tiles, several possible GPU-to-GPU coordination steps, one logits vector, one sampling decision, one KV-cache append, one detokenization fragment, and one network flush. If the user cancels at the wrong moment, the cancellation has to chase that chain backward so the scheduler can stop spending future iterations on a sequence nobody will read.

Important distinction: the model does not know the sentence it will write in advance. It has a probability distribution for the next token. The next token changes the context. That changes the next distribution. Fluent paragraphs emerge from repeated local choices conditioned on a very large learned state.

## Q13: How Do Tokens Become Text on Your Screen?

The serving worker **detokenizes** token ids back into text pieces. The serving process frames those pieces as stream events. The edge forwards them over the existing connection. The browser receives chunks, decodes bytes into text, updates the UI state, and paints.

This is why you can see half a sentence. The server is not waiting for the whole answer. It is flushing pieces as soon as policy and product logic allow.

The detokenizer has to be more careful than "id to string." Some tokens are word fragments. Some are byte fragments. A token boundary may not be a valid **UTF-8** boundary; UTF-8 is the common rule for turning bytes into characters. A **JSON** tool call, structured text often used for data, may be syntactically incomplete until several more tokens arrive. A citation marker may be withheld until the retrieval metadata is attached. A safety filter may delay release of a risky phrase until enough surrounding context exists. Streaming is therefore a small protocol inside the product: buffer when needed, flush when safe, and keep enough state to resume or stop cleanly.

Trace one fragment. The sampler chooses a token id. The detokenizer maps it to bytes or text. If those bytes complete a valid character, the stream gate can consider releasing it. If the fragment is inside a tool-call JSON object, the UI should not show it as prose. If it is a citation marker, the product needs source metadata. If it is risky text, the output check may need neighboring tokens before deciding. The user sees "typing." The system sees a sequence of release decisions.

There may be postprocessing in the stream:

- merge token fragments into valid UTF-8 text
- hold back unsafe partial output until checks pass
- parse tool-call JSON incrementally
- attach citations or retrieved document references
- redact internal control tokens
- account for generated tokens
- stop cleanly when the client cancels

The return path is easier than the forward path mathematically, but it is not trivial operationally. Long-lived streams consume connection slots. Mobile networks pause. Browsers sleep tabs. Users hit stop. Proxies time out. The serving system needs cancellation to propagate back to the scheduler so GPU work and KV memory are freed.

Cancellation is a reverse journey. The browser sends a close, abort, or stop signal. The edge marks the stream dead. The serving frontend marks the sequence cancelled. The scheduler removes it from the next iteration. The KV allocator releases its blocks. If a GPU kernel is already running, that iteration may finish, but the next one should not include the cancelled sequence. A bad cancellation path is expensive: users think they stopped generation, while the backend keeps paying for tokens no one will see.

The important LLM-specific point is that streaming is not merely "print as soon as possible." It is a controlled release path for a probabilistic generator. The product may need to hold text until bytes form valid characters, until tool JSON is complete enough to execute, until citation metadata is attached, or until a policy check has enough surrounding context. The best stream feels immediate to the user while still preserving these gates.

## Q14: What Else Is the Fleet Doing While You Wait?

Your request is one colored line in a giant moving map.

At the instant your stream is open, the fleet is not in one state. Some machines are loading model files. Some are warming GPU programs. Some are being drained for an update. Some are overloaded because a popular feature just sent a traffic spike. Some are serving long conversations with huge KV caches. Your request is one active record inside that moving system.

Trace one model update as it crosses your request.

A new model snapshot is copied into storage. The snapshot is not just weights. It includes tokenizer rules, chat format, number-format metadata, a split plan, and evaluation records. A few replica groups load it. Each GPU worker checks that it owns the right shard, moves that shard into GPU memory, and announces that it is warm. The router now has two worlds available: the old snapshot that already serves most users, and the new snapshot that should receive only a small slice.

Your prompt might be duplicated into **shadow traffic**: the old snapshot's answer is shown to you, while the new snapshot's answer is generated, logged, evaluated, and discarded. That lets the provider compare behavior without risking a visible regression. If the shadow path looks good, a **canary** begins: perhaps one percent of eligible traffic routes to the new snapshot. Now the change is real. Some users see the new behavior, and the fleet watches both answer-quality measurements and infrastructure measurements.

<FleetRolloutTrace client:visible />

The easy mistake is measuring only one side. A new snapshot can be safer and slower. A runtime change can preserve answer quality but make streamed tokens arrive less smoothly. A smaller-number build can fit more traffic but damage formatting in rare cases. A serious rollout treats every change as both a behavior experiment and a systems experiment.

Now trace one bad GPU inside that rollout. It does not crash. It merely runs slow. Your next token is ready on seven GPUs, but GPU 3 is late. The group waits. The next token waits. Time between streamed tokens rises even though the prompt, tokenizer, sampler, and model weights are fine. The symptom is human and vague: "the model feels slow." The owner may be hardware.

Now trace one traffic spike. A new feature sends many long prompts at once. Admission control starts rejecting low-priority long-context requests. The scheduler chunks prompt-reading work so answer streams do not freeze. KV-cache occupancy becomes the hard limit before raw arithmetic does. A provider that waits until users complain is too late; the fleet has to react when early signals cross guardrails.

The practical move is to keep the object in your hand. If the object is a snapshot, ask where it is stored, how it becomes warm, how it is tested, and how it rolls back. If the object is an active conversation record, ask what queue owns it, how much KV memory it holds, which replica group it entered, and when cancellation frees it. If the object is a token, ask which batch, kernel, communication step, sampler, stream gate, and browser paint it passed through. :::note[Same Symptom, Different Layer]

- Client/network: stream drops, so the answer stops mid-generation.
- Gateway: quota or login/session checks reject, so the request fails before model work.
- Router: bad placement, so the first answer piece arrives late.
- Tokenizer: unexpected token inflation, so cost and context use surprise you.
- Scheduler: queue instability, so waiting time spikes under load.
- KV cache: memory exhaustion or fragmentation, so requests are rejected, preempted, swapped, or slowed.
- GPU kernel: low utilization, so cost per token rises.
- Distributed shard: one slow or failed GPU, so the whole replica group stalls.
- Sampling/policy: bad stop logic or masking, so the answer is truncated, repetitive, or incorrectly refused.
- UI: poor stream handling, so text duplicates or formatting breaks.
:::

## Q15: How Does the Whole Thing Stay Stable?

At small scale, the serving question is "can one request finish?" At large scale, the question is "can the system keep answering while demand, prompt lengths, output lengths, and hardware health keep changing?"

Stability is not one switch. It is a set of feedback loops.

Trace one overloaded minute. A burst of long-context traffic arrives. The first signal is not necessarily errors. It may be rising queue depth, a growing prompt-reading backlog, KV cache filling, or the first token arriving later than usual. Admission control now has to decide whether another request should enter the system. It may accept, queue, reject, use a smaller model, reduce the maximum answer length, route to another region, or reserve capacity for higher-priority traffic. **Load shedding** is deliberate refusal: a clear failure for some users can be safer than letting everyone enter a queue so long that they time out and retry.

Now follow the feedback. If queues rise, users wait. If users wait, some cancel or retry. Retries add traffic. More active streams hold more KV cache. More KV pressure forces more rejection, swapping, preemption, or longer queues. This is the spiral the control loop is trying to prevent. Rate limits, token budgets, and max-output caps are therefore not merely billing rules. They are stability controls.

Autoscaling, adding more serving capacity automatically, is slower than admission because a warm LLM replica is not an ordinary web server. The system may need to copy hundreds of gigabytes of weights, verify shards, initialize GPU programs, allocate KV blocks, and then gradually route traffic to the new group. During the overloaded minute, the fastest controls are usually admission, routing, batching policy, output limits, and load shedding. New capacity helps later.

Trace a quality regression. A new snapshot passes ordinary tests. In canary, users start regenerating answers more often on finance questions with citations. The serving measurements look healthy: waiting time is fine, errors are low, GPUs are busy. The quality measurements are not healthy: cited sources no longer support some generated claims. A scoring model flags examples, humans audit a sample, and the team finds the model is over-summarizing retrieved tables. The rollback is a quality rollback, not an infrastructure rollback.

Trace a safety decision. A prompt enters. An input check may flag it. The model may still be allowed to answer if the safe response is educational or defensive. During generation, structured-output rules may prevent invalid tool calls. After generation, an output check may hold, redact, or replace text. If the system refuses too much, helpfulness falls. If it refuses too little, safety fails. Safety is a boundary under pressure, not a single yes/no box.

These loops share a discipline: every control must name both the signal it watches and the harm it may cause. Rejecting long prompts protects KV cache but harms users with legitimate long documents. Lowering max output protects decode capacity but may truncate good answers. Moving traffic to a smaller model improves availability but may reduce quality. Tightening a safety check reduces bad output but may over-refuse. The provider is not optimizing one number; it is maintaining a boundary.

The practical discipline is to map every symptom to the loop that could own it. A late first token may be admission, prefill, routing, or cold replicas. Bad citations may be retrieval, prompt packing, model behavior, or evaluation coverage. Unsafe output may be input checks, post-training, decoding constraints, output checks, or red-team coverage. Slow tokens may be a GPU, a communication step, a kernel, a queue, or a huge KV cache. The journey tells you which fixes could actually matter.

| Metric | Why it matters |
|---|---|
| Time to first token | captures queueing and prompt-reading pain |
| Inter-token latency | captures one-token-at-a-time generation health |

| Output tokens per request | predicts stream duration and decode load |
| KV-cache occupancy | memory admission bottleneck |
| Prefix-cache hit rate | whether repeated prompt prefixes are being reused |

## The Complete Loop

Months before your chat, training turned data into learned numbers, and post-training shaped those numbers into an assistant-like snapshot. When you press Enter, the browser sends encrypted bytes to a nearby service. The front door checks who you are, how much work you are asking for, and where the request should go.

The product then builds the real input, which is usually larger than the sentence you typed. It may include instructions, conversation history, retrieved evidence, tool descriptions, or image-derived numbers. The tokenizer turns that input into numbered text pieces. The scheduler places your active conversation beside other active conversations so a GPU step can advance many streams at once.

The model reads the input, saves reusable notes about the past, and then enters the one-piece loop. Layer after layer edits number tables. GPU memory feeds arithmetic circuits. If the model is split across chips, workers exchange partial results. At the end of the stack, the model has one raw score for every possible next piece. The runtime turns those scores into a weighted choice, picks one piece, appends it, streams it, paints it, and feeds it back into the next round.

The answer stops only when a stop condition wins: an end token, a length limit, a tool call, a policy rule, or your cancellation. Around that loop, the fleet is constantly deciding who enters, which replica runs, how memory is shared, when to shed load, which snapshot is safe to roll out, and which measurement proves the system is still healthy.

<CompleteLoopTrace client:visible />

So the answer on your screen is not "the model spoke." It is the visible edge of a loop that stayed coherent long enough to serve you: a frozen artifact was loaded correctly, your request was admitted, your prompt became vectors, KV cache remembered the prefix, GPU programs advanced the state, distributed workers synchronized, sampling chose one id, stream gates released safe text, and fleet measurements kept the whole system inside its guardrails. Understanding LLMs at scale means being able to point to the exact object at each layer and say what can change it, what can block it, and what evidence would prove it is healthy.
